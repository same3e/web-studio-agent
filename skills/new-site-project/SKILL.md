# New site project

## Before approval

1. Ensure the studio exists, then read every applicable file in `knowledge/` and inspect supplied client materials. Reuse facts already present.
2. Set `conversion-first` by default. Use `brand-led` only for an explicit image-led request, portfolio, fashion, art, architecture, or editorial work.
3. Create `projects/<clear-slug>/` and fill `CLIENT_BRIEF.md` from known facts. Gather service, audience, result, first-contact output, timing, price logic, geography, inclusions, proof, qualifications, process, constraints, prohibited claims, primary/alternative CTA, language, market, content, technology, requested visual style, and optional style modifiers.
4. Before creating any conversion-first H1, supporting copy, CTA, or proof points, create an internal `Offer facts` section with: primary service; target audience; client task or problem; concrete first-contact result; what is actually included in the first step; service mechanism; confirmed differences; proof; CTA and what happens after it; unconfirmed information; and prohibited claims. Classify every entry as `Confirmed facts` (usable in production copy), `Reasonable context` (use for understanding only, never turn into a concrete promise), or `Missing or unverified facts` (ask for it or retain only in internal documentation).
5. Run the fact gate before hero copy. Confirm that the core offer, first-contact result, CTA outcome, and every intended proof or differentiator are supported by `Confirmed facts`. If they are not, ask one short, critical batch of questions; offer answer options where helpful. Do not ask for information already in materials, fill gaps with invented advantages, raise Hero Copy Score with generic phrases, or start final copy approval. Never invent a number, guarantee, technology, qualification, review, price, advantage, or medical claim. Keep placeholders only in internal documentation.
6. Create an `Offer summary` after the fact gate: what we sell; who it is for; client result; mechanism; proof; first step.
7. Before selecting references, determine and record: (a) industry requirements, including the offer, conversion logic, trust signals, CTA pattern, and regulated-service constraints where relevant; (b) requested visual style family; and (c) optional style modifiers. If no visual style is requested, propose a suitable family and modifiers as a reversible direction, not as a confirmed decision. Read `knowledge/STYLE_INDEX.md` to interpret the vocabulary.
8. Read `knowledge/REFERENCE_INDEX.md` and select three deliberately distinct sets: relevant industry references for structure, conversion logic, trust elements, and CTA patterns; relevant style references for typography, composition, colours, shapes, spacing, motion, and UI treatment; and references matching both sets where available. Read only their linked `analysis.md` files as necessary. Do not duplicate a reference in project records just because it belongs to both sets; identify its roles instead.
9. Before concepts, show a short `Reference synthesis`: what is taken from industry references; what is taken from style references; how they combine; what must not be copied; and the likely usability or conversion risks. Industry-specific conversion logic remains the baseline; style is applied as a visual language, not as a substitute for offer clarity, accessibility, or trustworthy interaction.
10. Present exactly three genuinely different concepts: commercially safe, expressive, and experimental. Each must preserve the selected industry's conversion logic while applying the chosen visual language. Keep every hero hierarchy H1-first; never propose an eyebrow, kicker, overline, pretitle, pre-heading, or small hero label unless the user explicitly requests and separately approves it.

Each concept must contain a separate `Hero offer` with H1, supporting copy, CTA, one to three value-adding proof points when confirmed, facts used, facts requiring confirmation, and Hero Copy Score. Proof may be near the CTA, in supporting copy, or a compact trust element; its visual format follows the concept and need not be a card. Apply the whole-hero H1 specificity rule and the anti-template rule in `knowledge/COPY_RULES.md`; vary the approach by niche rather than swapping nouns in a stock formula. For regulated or multilingual projects, apply the medical-claim and multilingual-copy rules before presenting the offer.

Score factual reliability below the maximum whenever a key promise relies on an assumption. Award no differentiation/proof points for generic phrases such as “attentive doctors”, “modern approach”, or “comfortable atmosphere”. If the business has no confirmed unique advantage, build the offer honestly around the understandable process, audience, or first-step result instead of inventing uniqueness. Score conversion-first copy under `knowledge/COPY_RULES.md`. Do not approve a score below 8/10 unless the user explicitly selected `brand-led`.

Do not create final copy, implementation files, or an app before the user explicitly approves a concept.

## After approval

Create and fill `docs/APPROVED_CONCEPT.md`, `docs/SITEMAP.md`, `docs/PAGE_STRUCTURE.md`, `docs/COPY.md`, `docs/UI_SYSTEM.md`, `docs/IMPLEMENTATION_PLAN.md`, and a project `AGENTS.md`, using root templates. Preserve copy mode, core offer, promised result, supporting facts, approved hero copy, proof points, prohibited claims, Hero Copy Score, selected industry references, selected style references, style family, style tags, visual modifiers, and implementation guardrails in the relevant documents. Record assumptions instead of silently losing them.
